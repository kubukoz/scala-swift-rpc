# SSR

A Calico-inspired FRP UI framework for native-looking macOS apps written in Scala. A Swift `AppKit` host process renders the UI; a Scala child process owns application logic and the component tree. The two communicate over JSON-RPC (LSP framing) on stdin/stdout, with the wire protocol defined in Smithy.

Scala code lives under `package ssr`. The public tag DSL is `object ui` in `scala/dsl.scala` (`ui.vstack`, `ui.label`, etc.). Smithy-generated wire types live under `ssr.internal.protocol` â€” internal, users don't touch them.

## Layout

```
smithy/ui.smithy          Wire protocol (source of truth, both sides mirror it)
smithy-build.json         Drives smithy-cli; runs the Swift codegen plugin
codegen/swift-plugin/     Custom smithy-build plugin â†’ WireTypes.swift
scala/                    Component tree, FRP runtime, JSON-RPC client/server
  generated/              smithy4s output â€” gitignored, do not edit
swift/main.swift          AppKit host: spawns Scala, renders Nodes, emits events
  generated/              Swift wire types from smithy plugin â€” gitignored, do not edit
build/                    Build outputs (binaries, generated resources, .mode)
build.sh / run.sh         Build and launch
```

## Build & run

```bash
./build.sh            # codegen + Swift app + JVM Scala (default)
./build.sh --native   # codegen + Swift app + Scala Native binary
./run.sh              # launches Swift app; reads build/.mode to pick JVM vs native child
```

`build.sh` writes `build/.mode` so `run.sh` knows which Scala binary to spawn (`SCALA_APP_BIN` for native, `SCALA_APP_PATH` + `scala-cli run` for JVM). Two codegen passes run before the app builds:
- **Scala:** `cs launch ... smithy4s-codegen-cli` against `smithy/` â†’ `scala/generated/` (gitignored).
- **Swift:** `scala-cli publish local codegen/swift-plugin` installs the plugin to `~/.m2`, then `cs launch --contrib smithy-cli build` (driven by `smithy-build.json`) emits `build/smithy/source/swift-codegen/WireTypes.swift`, which is copied to `swift/generated/WireTypes.swift` (gitignored) and compiled alongside `swift/main.swift`.

Dependencies: `scala-cli`, `cs` (coursier), `swiftc`. Scala 3.8.3 pinned in `scala/project.scala`.

## Architecture

**Process model.** Swift is the parent. `AppDelegate` (`swift/main.swift`) launches Scala via `JSONRPCBridge`, wires stdout to a frame reader and stdin to a notification writer. All messages are JSON-RPC notifications (no request/response) framed with `Content-Length` headers.

**Protocol direction.**
- Scala â†’ Swift (`UiCommands`): `ui/mount`, `ui/patch`, `ui/window`, `ui/menu`, `ui/quit`
- Swift â†’ Scala (`UiEvents`): `event/click`, `event/input`, `event/frame`

Both sides must stay in sync with `smithy/ui.smithy`. Scala uses smithy4s codegen + `jsonrpclib-smithy4s`. Swift wire structs are generated by the custom plugin in `codegen/swift-plugin/` (`SwiftCodegenPlugin`); only the JSON-RPC method-name strings and handler wiring in `swift/main.swift` are hand-written.

**FRP component model** (Calico-inspired, `scala/`):
- `Component = SSR => Resource[IO, NodeBuilder]` â€” opaque type in `component.scala`. `SSR` (in `runtime.scala`) is the per-app context: `bus`, `emit`, and a `Signal[IO, WindowFrame]`.
- `Modifier[A]` typeclass (`modifier.scala`) composes children, attrs, events, signals, tuples, `Option`/`List`/`Resource`. Add new node-modifier types by giving a `Modifier` instance.
- `Attr[A]` supports static `:=` and reactive `<--` (with `Signal[IO, A]`). See `dsl.scala`.
- Reactive bindings (signals as text or attrs) apply the **current value synchronously** during build so the initial mount snapshot is correct, then subscribe via `discrete.evalMap(...).compile.drain.background`. Preserve this pattern when adding new reactive modifiers.
- `ui.{vstack, hstack, label, button, textfield}` are the tag constructors.

**App-level signals.** `App` exposes `window: Signal[IO, SetWindowInput]` and `menu: Signal[IO, List[MenuItem]]` (see `main.scala`). `Main.run` reads each via `getAndDiscreteUpdates`, sends the initial value before `ui/mount`, then drives subsequent `ui/window` / `ui/menu` notifications from the update stream in a background fiber.

**Mount/patch lifecycle** (`node.scala`, `main.scala`):
1. Build the component tree as a `Resource[IO, NodeBuilder]`. Each `NodeBuilder` gets a fresh id from `IdGen`.
2. `root.snapshot` produces a full `Node` tree; send once via `emit.mount(tree)`.
3. `root.markMounted` flips the per-node `mounted` flag. **Only after this** do `setText` / `setValue` emit `ui/patch` notifications. Mutations before mount are silent (just update local refs) so the initial snapshot stays consistent.
4. Swift's `Renderer` indexes views by id on mount; `patch` notifications look up the view and apply `setText` / `setValue`.

**Event flow.** Swift handlers (`ClosureButton`, `TextFieldDelegate`, menu items) send `event/click` / `event/input` with the node id. Scala's `eventService` forwards to `EventBus`, which dispatches by id to handlers registered via `OnEvent` modifiers (`onClick`, `onInput`). `Ctx` carries the bus, the emit client, and a `Signal[IO, WindowFrame]` for the current window geometry.

**Window state.** `event/frame` notifications update the frame signal and persist to `~/.local/state/ssr/window.json` (see `App.loadFrame` / `saveFrame`). On startup, the saved frame becomes the initial `SetWindowInput`.

## Conventions / gotchas

- **Don't edit `scala/generated/`** â€” it's regenerated by `build.sh` and gitignored.
- **Don't emit patches before `markMounted`.** Setters check the flag; if you bypass this you'll desync the Swift view index.
- **Don't edit `swift/generated/`** â€” regenerated by `build.sh` and gitignored. Add Swift wire types by extending `SwiftCodegenPlugin` (e.g., new shape kinds, conformances) and rerunning `build.sh`.
- **Adding a new operation:** update `smithy/ui.smithy`, then add the handler + matching method-name string in `swift/main.swift`'s `enum Methods`. Struct generation is automatic; routing is not.
- **Editing the Swift plugin requires republishing:** `build.sh` runs `scala-cli publish local codegen/swift-plugin` every build, so iteration is `./build.sh`. If you call `cs launch --contrib smithy-cli build` directly, you must publish the plugin first or smithy-build will fail to resolve `ssr:swift-codegen_3:0.1.0-SNAPSHOT`.
- **Reactive modifier pattern:** synchronous initial apply + background discrete subscription. See `Modifier[Signal[IO, String]]` and the `AttrSignalPair` given for the template.
- The frame-codec (`App.frameCodec`) is derived from the smithy schema via `CirceJsonCodec.fromSchema` â€” use this approach if persisting other wire types.
- All RPC is notifications (no responses). If you need request/response, that's a protocol-level change in `ui.smithy` and a rework of `JSONRPCBridge` on the Swift side.
