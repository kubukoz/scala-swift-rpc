# SSR

A Calico-inspired FRP UI framework for native-looking macOS apps written in Scala. A Swift `AppKit` host process renders the UI; a Scala child process owns application logic and the component tree. The two communicate over JSON-RPC (LSP framing) on stdin/stdout, with the wire protocol defined in Smithy.

Scala code lives under `package ssr`. The public tag DSL is `object ui` in `scala/dsl.scala` (`ui.vstack`, `ui.label`, etc.). Smithy-generated wire types live under `ssr.internal.protocol` — internal, users don't touch them.

## Layout

```
build.sbt                 Top-level sbt build (modules + root tasks + publishing)
project/                  sbt meta-build (plugins, SwiftCodegenRunner helper)
smithy/                   Wire protocol (source of truth, both sides mirror it)
codegen/swift-plugin/     Custom smithy-build plugin → WireTypes.swift (sbt module)
scala/                    Component tree, FRP runtime, JSON-RPC client/server (sbt projectMatrix)
sbt-ssr/                  The packaging plugin (Scala 2.12): bundles a user app + host into a .app
scala/testkit/            ssr-testkit — headless in-process test harness (projectMatrix, published)
scripts/ssr-uitest.swift  AX-based real-host UI driver (dump / read / click)
swift/main.swift          AppKit host: spawns Scala, renders Nodes, emits events
  generated/              Swift wire types from smithy plugin — gitignored, do not edit
build/                    Build outputs (ssr-app binary, JVM shim)
```

## Build & run

```bash
sbt runJVM         # build everything and launch with the JVM Scala child (stdio)
sbt runNative      # build everything and launch with the Scala Native child (stdio subprocess)
sbt -Dssr.ffi=true runFfi   # embedded: link the Native app into the host as a static lib, no subprocess (FFI transport)
sbt swiftBuild     # only build the Swift host (cacheable)
sbt swiftCodegenOutput  # only regenerate WireTypes.swift (cacheable)
sbt ssr3/compile        # compile the library, JVM axis (ssrNative3 for native)
sbt demos3/compile      # compile the demos, JVM axis (demosNative3 for native)
```

`sbt-projectmatrix` exposes `ssr3` / `ssrNative3` (library) and `demos3` / `demosNative3` (demos) axes that share sources in `scala/lib` and `scala/demos`. Smithy4s codegen for Scala is driven by the `smithy4s-sbt-codegen` plugin — outputs go to each axis's `target/<axis>/src_managed/` and are never committed.

Swift codegen + Swift build are sbt tasks (`swiftCodegenOutput`, `swiftBuild`) cached via `FileFunction.cached` with `FilesInfo.hash`. Inputs:
- `swiftCodegenOutput` — `smithy/**/*.smithy` + the `swiftCodegen` module's runtime classpath. Drives `ssr.codegen.SwiftCodegenMain` in a forked JVM (so the plugin loads via SPI without polluting sbt's classloader).
- `swiftBuild` — all `.swift` files (including the generated `WireTypes.swift`) + `swift/Package.swift`. Runs `swift build -c release` and copies `ssr-host` to `build/ssr-app`.

Dependencies: `sbt`, `swiftc`. Scala 3.8.3 pinned in `build.sbt`.

## Publishing & distribution

The build publishes to Maven Central under **`com.kubukoz`** via **sbt-typelevel** (version is git-tag-derived from `tlBaseVersion`, no `version :=`). Only three modules publish: `ssr3` (library, JVM), `ssrNative3` (library, native), and `plugin` (`sbt-ssr`, Scala 2.12). `root`, `demos`, and `swiftCodegen` are `NoPublishPlugin`.

CI (`.github/workflows/ci.yml`, `clean.yml`) is **generated by sbt-typelevel** — never hand-edit it; change `build.sbt` and run `sbt githubWorkflowGenerate`, or `githubWorkflowCheck` to verify freshness. A build preamble apt-installs the Scala Native toolchain on the ubuntu runner (native `test` links a real binary); a postamble runs `plugin/scripted`. The header check, scalafmt check, and fatal warnings are turned **off** (`tlCiHeaderCheck` / `tlCiScalafmtCheck` / `tlFatalWarnings := false`) — the existing Scala 3 sources predate headers and there's no `.scalafmt.conf`.

`host-release.yml` is **hand-maintained** (not tl-generated): on a `v*` tag it builds `ssr-app` on macos-14 via `sbt swiftBuild` and attaches it to the GitHub release. This is the asset `sbt-ssr`'s default host source fetches, keeping the host binary and Maven artifacts on the same tag.

**`sbt-ssr` (the packaging plugin, `sbt-ssr/`).** `SsrPlugin` (Scala 2.12, `noTrigger` — enable explicitly) assembles a user's app into `<AppName>.app` via the `ssrPackage` task. It lives on its own axis and does **not** `dependsOn` the Scala 3 `ssr` module — it only orchestrates file layout. Key settings: `ssrChildBinary` (the Scala child → staged as `Contents/Resources/ssr-child`), `ssrAssetsDir` (→ `Contents/Resources/assets/`), `ssrHostSource` (how to get the Swift host). `SsrHostSource` has three tiers: `Release(tag)` (default — download `ssr-app` from a GitHub release, cached under `target/ssr-host/`), `LocalPath(file)` (a host binary already on disk), `LocalBuild(pkg, product)` (`swift build -c release`). The `ssr-child` / `assets/` names are the hard contract the host's self-locating fallback reads (see the "self-locating" note under Conventions). Scripted test: `sbt plugin/scripted` (fixture in `sbt-ssr/src/sbt-test/ssr-package/basic`, uses the `LocalPath` tier so it needs neither network nor swiftc).

## Architecture

**Process model.** Swift is the parent. `AppDelegate` (`swift/main.swift`) launches Scala via `JSONRPCBridge`, a thin wrapper around `ChimeHQ/JSONRPC`'s `JSONRPCSession` (id allocation, pending-response correlation), with `Content-Length` framing supplied by `ChimeHQ/LanguageServerProtocol`'s `withMessageFraming`. Messages can be notifications **or** request/response in either direction.

**Two transports.** The JSON-RPC layer is transport-agnostic: `Main.run(factory, transport)` (`scala/lib/main.scala`) is the shared bootstrap, and a `Transport` (`scala/lib/transport.scala`) supplies just the raw framed-byte channels (`fromHost: Stream[IO, Byte]`, `toHost: Pipe[IO, Byte, Nothing]`); LSP framing + the `FS2Channel` + mount/patch sit above it, identical for both.
- **stdio (default).** `Transport.stdio` = `fs2.io.stdin/stdout`. Scala runs as a child process; Swift spawns it and pipes (`JSONRPCBridge(executable:arguments:)`, `makeChannel` over `Pipe`s). This is what `runJVM` / `runNative` / the packaged `.app` use.
- **FFI (embedded, Native only).** The whole Scala app is linked into the Swift binary as a **static library** exporting `ssr_init` — no subprocess. This is the single-binary path toward iOS / the App Store (where spawning children isn't allowed). Messages flow through two shared-memory **SPSC ring buffers** (`scala/lib-native/ffi.scala`, mirrored in Swift's `Ring`/`FfiTransport` under `#if SSR_FFI`). Selected at runtime by `SSR_FFI=1`; built with `-Dssr.ffi=true`. See the FFI subsection below.

**FFI transport details** (`scala/lib-native/ffi.scala`, `swift/main.swift`, gated `-Dssr.ffi=true` / `#if SSR_FFI`).
- **Rings.** Two malloc'd (native-heap, never GC-moved) SPSC ring buffers. Layout both sides agree on: `Int64 capacity` @0, `Int64 head` @8, `Int64 tail` @16, `data[capacity]` @24; live region `[head, tail)`, slot `i & (capacity-1)`. `inRing` = host→Scala (Swift produces), `outRing` = Scala→host (Swift consumes). `ssr_init()` allocates them, boots the app, returns a handle = `{inRing*, outRing*}` that Swift reads at offsets 0/8. Bytes carried are LSP-framed JSON-RPC — same as stdio. Poll-based (1ms idle sleep on both ends), no callbacks.
- **Runtime bootstrap.** Static libs (unlike dylibs) get **no** load-time constructor for Scala Native's `ScalaNativeInit` (GC/threads), so Swift calls the exported `ScalaNativeInit()` itself once before `ssr_init()` (see `FfiTransport.init`). For `ScalaNativeInit` to be *emitted*, the demos-native axis must have **no mainClass** in FFI mode (`entry.fold(genLibraryInit)(genMain)` in the toolchain) — build.sbt unsets `mainClass` + `discoveredMainClasses` under `-Dssr.ffi=true`.
- **GC-safepoint gotcha (critical).** The foreign (Swift) thread that calls `ssr_init` gets registered as a *Managed* GC mutator, then returns into AppKit's run loop and blocks in `mach_msg` forever — never polling a safepoint, which **deadlocks stop-the-world GC** (symptom: "Waiting for N thread(s) to reach safepoint" spam, no window). Fix: `Ffi.boot` switches the calling thread to **Unmanaged** (`GCState.setMutatorThreadState(1)`, an `@extern` binding to `scalanative_GC_set_mutator_thread_state`) right before returning. The app itself runs on Cats Effect's own worker threads, which poll safepoints normally. `unsafeRunAndForget` is fine — CE 3.7 Native uses a real multi-threaded work-stealing pool that drives timers without the main thread.
- **App entry.** An app object mixes in `SsrFfiApp` (native-only, `ffi.scala`) — the FFI analogue of `SSRApp` — and declares `@exported("ssr_init") def ssrInit(): Ptr[Byte] = boot()`. `boot()` forwards to `Ffi.boot(render)`. The demo is `scala/demos-native/LandmarksFfi.scala`, reusing `LandmarksMain.render`.
- **Build/run.** `sbt -Dssr.ffi=true runFfi` builds the static lib (`demosNative3/nativeLink` → `libssr-demos.a`), links it into the host (`swift build` with `SSR_FFI_LIB=<abs .a>` → Package.swift adds `-force_load` + `-DSSR_FFI`), and launches with `SSR_FFI=1`. The default (no `-Dssr.ffi`) host builds with no FFI symbols referenced, so the stdio path is unaffected.

**Protocol direction.**
- Scala → Swift (`UiCommands`): notifications `ui/mount`, `ui/patch`, `ui/replaceChildren`, `ui/window`, `ui/menu`, `ui/quit`; requests `ui/openPanel`
- Swift → Scala (`UiEvents`): notifications `event/click`, `event/input`, `event/toggle`, `event/frame`

Both sides must stay in sync with `smithy/ui.smithy`. Scala uses smithy4s codegen + `jsonrpclib-smithy4s`. Swift wire structs are generated by the custom plugin in `codegen/swift-plugin/` (`SwiftCodegenPlugin`); only the JSON-RPC method-name strings and handler wiring in `swift/main.swift` are hand-written.

**FRP component model** (Calico-inspired, `scala/`):
- `Component = SSR => Resource[IO, NodeBuilder]` — opaque type in `component.scala`. `SSR` (in `runtime.scala`) is the per-app context: `bus`, `emit`, and a `Signal[IO, WindowFrame]`.
- `Modifier[A]` typeclass (`modifier.scala`) composes children, attrs, events, signals, tuples, `Option`/`List`/`Resource`. Add new node-modifier types by giving a `Modifier` instance.
- `Attr[A]` supports static `:=` and reactive `<--` (with `Signal[IO, A]`). See `dsl.scala`.
- Reactive bindings (signals as text or attrs) use `signal.getAndDiscreteUpdates`: apply the returned initial value synchronously during build so the mount snapshot is correct, then subscribe to `updates.evalMap(...).compile.drain.background`. Preserve this pattern when adding new reactive modifiers. Don't use `signal.get` + `signal.discrete` separately; that races with updates and double-applies the current value.
- `ui.{vstack, hstack, label, button, textfield}` are the tag constructors.

**App-level signals.** `App` exposes `window: Signal[IO, SetWindowInput]`, `menu: Signal[IO, List[MenuItem]]`, and `statusItem: Signal[IO, Option[SetStatusItemInput]]` (default `constant(None)`), plus a one-shot `activationPolicy: ActivationPolicy` (default `REGULAR`; `ACCESSORY` = menu-bar agent, no Dock icon) — see `main.scala`. `Main.run` reads each signal via `getAndDiscreteUpdates`, sends the initial value before `ui/mount`, then drives subsequent `ui/window` / `ui/menu` / `ui/statusItem` notifications from the update stream in a background fiber; `activationPolicy` is sent once before mount. `SetWindow` has a `visible: Boolean` (absent = visible) so a menu-bar app can hide its window (`window.orderOut`) until asked to show it.

**Status item & menus.** `ui/statusItem` (`SetStatusItem`) creates/updates/removes an `NSStatusBar` item: a `menu` builds a dropdown, else the button fires `event/click` with `id`. `MenuItem` gained `enabled: Boolean` (absent = enabled) and `style: MenuItemStyle` (`HEADER` = bold + disabled title line) for status-item dropdowns; the status-item menu sets `autoenablesItems = false` so disabled/header items stay disabled.

**High-level menu DSL** (`statusitem.scala`). Rather than hand-building `List[MenuItem]` with string ids and separately registering bus handlers (the pre-`onClick` model), the `menu` DSL describes entries with click handlers inline — same ergonomics as `onClick`. `menu.item(title, enabled?, key?)(onClick)`, `menu.text`, `menu.header`, `menu.separator`, `menu.submenu(title)(children*)`, and `menu.menu(title)(children*)` (a top-level app menu; same shape as a submenu). All build a `StatusMenuItem`. Two consumers, both needing a `given SSR`:
- `ssr.statusItem(title, items*)` (or a `Signal[IO, List[StatusMenuItem]]` overload) — the menu-bar status item. Returns `Resource[IO, Signal[IO, Option[SetStatusItemInput]]]` for `App(statusItem = ...)`.
- `ssr.appMenu(menus*)` (or a signal overload) — the application menu bar. Returns `Resource[IO, Signal[IO, List[MenuItem]]]` for `App(menu = ...)`. Replaces the old raw-`MenuItem` + hardcoded `App.QuitMenuId` pattern (quit is now just `menu.item("Quit", key = Some("cmd+q"))(emit.quit().void)`). `QuitMenuId` + its `main.scala` bus registration remain only as back-compat for hand-built raw menus.

Both are built on the private `MenuRender` helper: `render` assigns fresh ids (via `idGen`) and registers handlers on the `EventBus` for the render's Resource scope; `driven` drives a reactive signal through it, swapping each render's handler batch via a `Hotswap` (releasing the previous atomically) and pushing the mapped result to an output `SignallingRef`. No keyed diffing — re-renders just re-send the whole menu, which is cheap. Behavior covered by `scala/lib-test/StatusItemTests.scala` and `NotificationsTests.scala` (munit + munit-cats-effect; the `ssr` module's only test source dir, `Test / scalaSource := scala/lib-test`, runs on both JVM and Native under `sbt test`). Notification tests use a recording `UiCommandsGen.Default[IO](IO.stub)` subclass overriding just the notification ops.

**Notifications.** `SSR.notifications: Notifications` (see `notifications.scala`) wraps `UNUserNotificationCenter` via four commands — `requestAuth` (`@jsonRpcRequest`), `setCategories`, `notify` (generates as `emit._notify` — smithy4s reserved-name avoidance), `scheduleReminder` (repeating `UNCalendarNotificationTrigger` at hour:minute) — and routes taps back through `event/notificationTapped` → `EventBus`, keyed by `categoryId` (same mechanism as `onClick`/`QuitMenuId`). `notifications.onTapped(categoryId)(handler)` registers a handler for its Resource scope; `actionId` is an action's id or `"default"` for a body tap. Request ops must declare both `input` and `output` (the Swift codegen plugin enforces it) — `RequestNotificationAuth` has an explicit empty `input := {}`.

**Progress.** `ui.progress` renders an `NSProgressIndicator`; bind `attrs.fraction: Attr[Option[Double]]` — `Some(f)` (clamped to [0,1]) = determinate, `None` = indeterminate. The fraction rides the node's `value` string (`""` = indeterminate).

**Mount/patch lifecycle** (`node.scala`, `main.scala`):
1. Build the component tree as a `Resource[IO, NodeBuilder]`. Each `NodeBuilder` gets a fresh id from `IdGen`.
2. `root.snapshot` produces a full `Node` tree; send once via `emit.mount(tree)`.
3. `root.markMounted` flips the per-node `mounted` flag. **Only after this** do `setText` / `setValue` emit `ui/patch` notifications. Mutations before mount are silent (just update local refs) so the initial snapshot stays consistent.
4. Swift's `Renderer` indexes views by id on mount; `patch` notifications look up the view and apply `setText` / `setValue`.

**Event flow.** Swift handlers (`ClosureButton`, `TextFieldDelegate`, menu items) send `event/click` / `event/input` with the node id. Scala's `eventService` forwards to `EventBus`, which dispatches by id to handlers registered via `OnEvent` modifiers (`onClick`, `onInput`). `Ctx` carries the bus, the emit client, and a `Signal[IO, WindowFrame]` for the current window geometry.

**Window state.** `event/frame` notifications update the frame signal and persist to `~/.local/state/ssr/window.json` (see `App.loadFrame` / `saveFrame`). On startup, the saved frame becomes the initial `SetWindowInput`.

## Testing SSR apps

Two tiers, headless-first. Both let you drive an app without babysitting clicks by hand (see the ui-testing memory for why real event delivery matters).

**In-process harness (`ssr-testkit`, `scala/testkit/`)** — the default, 90% case. `TestHarness.of(factory): Resource[IO, TestHarness]` builds the component tree through the *real* runtime path (`Component.build` + `markMounted`, reactive window/menu/status background fibers) but swaps a **recording `UiCommands`** (`RecordedCommands`, captures `mount`/`patch`/`setWindow`/`setMenu`/`setStatusItem`/`quit` into a `Recorded`) in for the JSON-RPC channel. No Swift host, no window, no stolen focus, cross-platform (JVM + Native under `sbt testkit3/test` / `testkitNative3/test`). Drive events (`click`/`input`/`toggle`/`setFrame` — routed straight into the `EventBus` exactly as `eventService` does); query the live tree (`tree`, `textOf`/`valueOf` by id, `findByText`/`findAll`, `clickText`) or the recorded protocol traffic (`recorded`). Drive methods `settle` after firing — poll the `Node` snapshot to a fixed point (stable for N consecutive reads) — because signal→setText propagation runs in **background fibers**, so a naive read-after-fire races ahead of the update. Published as a Test-scope dep (`com.kubukoz %% ssr-testkit`); `dependsOn(ssr)`. Node ids are runtime-assigned (`n1`…); prefer by-text/by-tag matching in tests. Tests live in `scala/testkit-test/`.

**Real-host AX driver (`scripts/ssr-uitest.swift`)** — the 10% "does it truly render + click" smoke test. Drives the actual running `ssr-app` process via the macOS Accessibility API. Subcommands: `dump <proc>` (AX tree), `read <proc> <role> <title>` (an element's value), `click <proc> <role> <title> [--real] [--drag]`. **The process name is `ssr-app`, not `ssr-host`** (`build/ssr-app` is what `runJVM`/`runNative` launch; a stray `ssr-host` may belong to another app). Default click = `AXPress` — **focus-free** (safe to run in the background, won't steal focus) but bypasses AppKit event delivery. `--real` posts a physical `CGEvent` click at the element center — highest fidelity, catches hit-test/gesture bugs, but **requires the app frontmost** so it briefly steals focus; `--drag` (implies `--real`) nudges the cursor between down/up. Matches title against AXTitle then AXValue (SSR labels expose text as AXValue). Needs Accessibility permission for the driving terminal. Supersedes the old `ax-click`/`cg-click`/`cg-drag-click`/`dump-ax` scripts.

## Conventions / gotchas

- **Don't edit smithy4s output** — written to each axis's `target/<axis>/src_managed/main/smithy4s/`, regenerated by sbt on every compile.
- **Don't emit patches before `markMounted`.** Setters check the flag; if you bypass this you'll desync the Swift view index.
- **Don't edit `swift/generated/`** — regenerated by `sbt swiftCodegenOutput` and gitignored. Add Swift wire types by extending `SwiftCodegenPlugin` (e.g., new shape kinds, conformances) and rerunning the task.
- **Adding a new operation:** update `smithy/ui.smithy`. For notifications, register a handler in `swift/main.swift` next to the existing `bridge.on(Methods.xxx)` calls. For request/response ops (`@jsonRpcRequest`) the plugin emits a typed `bridge.on<Op>` extension — just call it with a closure returning the typed output. Struct + method-name + per-op extension generation is automatic; only the handler body is hand-written.
- **Editing the Swift plugin:** changes to `codegen/swift-plugin/SwiftCodegenPlugin.scala` are picked up automatically on the next `sbt swiftCodegenOutput` — sbt rebuilds the `swiftCodegen` module and hashes its classpath as part of the codegen cache key. No publish step.
- **Reactive modifier pattern:** `signal.getAndDiscreteUpdates` returns `(initial, updates)` atomically — apply `initial` synchronously, then drain `updates` in the background. See `Modifier[Signal[IO, String]]` and the `AttrSignalPair` given for the template. Don't use `signal.get` + `signal.discrete` separately; that races with updates and double-applies the current value.
- The frame-codec (`App.frameCodec`) is derived from the smithy schema via `CirceJsonCodec.fromSchema` — use this approach if persisting other wire types.
- **Bidirectional request/response is supported.** Tag a smithy op with `@jsonRpcRequest("ui/foo")` for request/response, `@jsonRpcNotification("ui/foo")` for fire-and-forget. On the Scala side smithy4s generates `def foo(...): IO[FooOutput]` automatically. On the Swift side the codegen plugin emits a typed `bridge.onFoo { input in … return FooOutput(...) }` extension on `JSONRPCBridge`. `JSONRPCSession` owns id allocation and pending-response correlation; we only provide handler bodies.
- **Swift host is a SwiftPM target** (`swift/Package.swift`). The `swiftBuild` sbt task wraps `swift build -c release` with content-hash caching. For tighter loops you can still `cd swift && swift build` directly. Dependencies: `ChimeHQ/JSONRPC`, `ChimeHQ/LanguageServerProtocol` (used only for `DataChannel.withMessageFraming`).
